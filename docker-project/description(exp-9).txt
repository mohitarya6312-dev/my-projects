Project: Automated CI/CD Pipeline for a Dockerized React App
1. Technologies Used
React.js: The core library for building the application's user interface.

Docker: Used to containerize the application, featuring a multi-stage build for an optimized, lightweight final image.

Nginx: Acts as the high-performance web server serving the static React build within the production container.

GitHub Actions: The CI/CD platform used to orchestrate the entire automated workflow from code push to deployment.

Docker Hub: The public container registry used for storing and distributing the final, production-ready Docker image.

Node.js (with npm): Provides the runtime environment for building the React app (npm run build) and installing exact dependencies (npm ci).

YAML: The configuration language used to define the GitHub Actions workflow steps.

2. Project Overview
This project establishes a complete Continuous Integration and Continuous Deployment (CI/CD) pipeline for a React application, fully automated using GitHub Actions and containerized with Docker.

The workflow demonstrates a modern DevOps approach, automating the entire journey from a code push to a deployable artifact. Specifically, any commit to the react-project/Experiment-7/ directory automatically triggers the pipeline. This process installs dependencies, builds the React application, packages the static files into a production-optimized Docker image, and publishes that final image to Docker Hub.

The primary objectives were to apply DevOps principles, implement a robust automated workflow, optimize container builds using multi-stage Dockerfiles, and manage credentials securely using GitHub Secrets.

3. Project File Structure
The project is organized within a monorepo. The CI/CD workflow is defined in .github/workflows/build.yml. The React application, along with its specific Dockerfile, is nested within react-project/Experiment-7/, allowing the workflow to granularly target changes to that specific project.

4. CI/CD Workflow Structure
The application's build and deployment are orchestrated by the build.yml workflow file.

Multi-Stage Dockerfile The Dockerfile is a critical component that leverages a multi-stage build to create a minimal, secure, and efficient production image.

Stage 1 (Build): A Node.js container is used as a temporary build environment. It installs dependencies using npm ci and compiles the production-ready static assets.

Stage 2 (Production): A lightweight nginx:alpine image serves as the final base. Only the compiled assets from the 'build' stage are copied over, ensuring the final image contains no build-time dependencies, source code, or vulnerabilities.

Workflow Triggers and Defaults The build.yml workflow is precisely configured to run only when changes are pushed to the react-project/Experiment-7/ path on the main branch. This path-based trigger is essential for a monorepo. It also includes a workflow_dispatch trigger for manual execution and sets a working-directory default, ensuring all commands execute from within the correct project folder.

Automated Build This "Continuous Integration" (CI) phase checks out the repository, configures the correct Node.js version, and executes the build process. Using npm ci guarantees a reproducible build by installing exact dependency versions from the lockfile, followed by npm run build to create the static site.

Docker Build & Push to Registry This "Continuous Deployment" (CD) phase, conditional on pushes to the main branch, securely logs into Docker Hub using credentials stored in GitHub Secrets. It then builds the Dockerfile, tags the resulting image, and pushes it to the Docker Hub registry, making the new version available for deployment.
